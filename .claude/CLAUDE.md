# Global Preferences

## Code Workflow
- Always explain what you plan to do before making changes
- Always explain how you will do it
- Always show what will be changed
- Wait for confirmation before executing any code changes
- Never edit code without explicit approval

## Code Style
- Keep implementations minimal
- Prefer editing existing files over creating new ones
- Never create files unless absolutely necessary
- No emojis unless explicitly requested
- Clean, concise code without redundancy
- Never assume enum values, choices, or constants — only use what has been explicitly stated
- Never add docstrings or module-level comments unless explicitly stated
- Avoid inline comments unless explicitly asked; if used, keep to a few words max — never use comments to explain logic or intent
- When modifying code, always comment out the old code instead of deleting it

## Communication Style
- Short and concise responses
- Professional and objective tone
- Focus on facts and problem-solving
- No unnecessary praise or validation
- No over-the-top validation phrases

## Technical Preferences
- Use specialized tools (Read/Edit/Write) instead of bash commands for file operations
- Always confirm understanding before implementing features
- Ask clarifying questions when requirements are ambiguous
- Direct answers without over-explaining
- Never install packages — update requirements.txt instead, then inform the user to install after all other changes are done

## Import Style
- Use relative imports from project root when possible
- Avoid redundant module prefixes

## General
- Only use TodoWrite for complex multi-step tasks
- Mark todos as completed immediately after finishing
- Don't batch todo completions
- Always ignore the `.sketch` folder unless explicitly asked about it
- Never include Co-Authored-By in commit messages unless explicitly asked
